\begin{abstract}
Deploying vision based autonomous control on embedded hardware requires careful design of insight and actuation pipelines. Unlike approaches that rely on off board compute, GPU acceleration, or multiple sensors, this system achieves sub 100ms autonomous navigation on a Raspberry Pi 4 using only a single USB camera demonstrating that tight coordination between a 15 FPS vision pipeline and 50Hz BLE command transmission is achievable within a 78\% CPU budget. Using C++17, OpenCV, and Bluetooth Low Energy, the system achieves 15FPS tracking with 50Hz command transmission and sub-100ms latency. Key contributions include dual-method car detection (motion-primary, color-fallback), HSV-based centerline path extraction, and proportional steering control. The multi-threaded architecture demonstrates robust autonomous navigation with graceful fault tolerance and configuration-driven tuning. Implementation supports macOS, Linux, and Raspberry Pi platforms.
\end{abstract}

\section{Introduction}

Autonomous systems require real-time perception and control. Small-scale platforms like RC cars on marked tracks provide an ideal testbed for vision-based control, multi-threaded architecture, and wireless communication in resource-constrained environments. Unlike traditional autonomous driving (requiring LiDAR and multiple sensors), our system uses a single camera with efficient algorithms to achieve robust track-following.

This paper presents the design, implementation, and validation of a production-ready autonomous RC car controller. We demonstrate that robust autonomous control is achievable with modest computational resources through: (1) motion-based primary detection with color-based fallback, (2) centerline-guided path following, (3) multi-threaded orchestration, and (4) configuration-driven parameter tuning. The system targets macOS, Linux, and Raspberry Pi platforms.

The main challenge is not perception or actuation, but their real-time coordination under a shared computational budget. A tracking pipeline that consume available CPU leaves insufficient headroom for reliable BLE transmission, while a reduced control rate introduces latency that destabilizes steering. This work addresses that tension directly through thread isolation, frame rate limiting, and lazy centerline updates, sustaining 50 Hz command delivery alongside stable 15 FPS tracking, with no commands dropped over a 45 second autonomous run on Raspberry Pi 4.

\section{System Architecture}

\subsection{Hardware Components}

The system comprises three hardware layers:

\textbf{Computation Platform:} Macintosh/Linux desktop for development. Raspberry Pi~4 Model~B (8GB RAM) for embedded deployment. Both support 1280×720 at 30 FPS USB camera input and native BLE 5.0 connectivity.

\textbf{Sensing:} USB camera with configurable resolution (640×480 to 1280×720) and frame rate (10-30 FPS). The system adapts to camera initialization failures by attempting multiple backends (V4L2, GStreamer, direct OpenCV fallback).

\textbf{Actuation:} BLE-controlled RC car with maximum velocity $v_{\max} = 0.55~\text{m/s}$ (measured via 3~m distance test over 10 runs, average time 5.47~s). Currently operates at 40\% maximum speed ($v_{\text{active}} = 0.22~\text{m/s}$). Car model expects: light on/off flag, forward speed (0--255), left steering (0--100), right steering (0--100), and device identifier for message routing.

\textbf{Track Specifications:} Oval closed-loop track with perimeter 540~cm, height 250~cm, and width 130~cm. Red/white boundary markers enable centerline detection.

\subsection{Software Architecture}

Five core modules interact through thread-safe queues and atomic variables, as shown in Figure~\ref{fig:1}:

\textbf{Camera Capture:} Manages camera initialization, resolution/FPS negotiation, and continuous frame acquisition at native camera FPS. Implements fallback logic for device enumeration.

\textbf{Car Detector:} Implements dual-method detection with motion-based primary (MOG2 background subtraction, threshold 80, learning rate 0.003) and color-based fallback (HSV red detection across six hue ranges). Both methods apply morphological filtering and area constraints (200--20000 pixels).

\textbf{Boundary Detector:} Constructs track obstacle maps via HSV analysis: red pixels (hue 0--12°, 168--179°) form track boundaries; white pixels (saturation 0--70, value 180--255) mark interior markers. Convex hull of red pixels defines track ROI. Centerline extraction applies distance transform and skeleton extraction on flood-filled free space.

\textbf{Control Generator:} Computes steering via two modes. Ray casting mode (when centerline unavailable) casts three rays ($-60\degree$, $0\degree$, $+60\degree$ relative to heading) and generates evasive steering if closest ray distance $<100$ pixels. Pure pursuit mode (when centerline available) calculates offset between car position and nearest centerline point, applying proportional steering: $\text{steering} = k_p \times \text{offset}$ where $k_p$ is tuned via configuration.

\textbf{BLE Handler:} Manages device scanning, connection retry (up to 3 attempts with 1 second delays), command formatting, and transmission at 50 Hz. On shutdown, sends 10 STOP commands (500 ms total) before disconnection.

\textbf{Control Orchestrator:} Synchronizes threads, manages frame queues (5-frame circular buffer), enforces 15 FPS tracking rate (67 ms sleep when needed), and publishes latest ControlVector to BLE thread.

\subsection{Multi-Threaded Execution}

Four threads execute concurrently:
\begin{enumerate}
  \item \textbf{Camera thread:} Continuous frame capture and queue push
  \item \textbf{Tracking thread:} 15 FPS detection, boundary analysis, control vector generation
  \item \textbf{BLE thread:} 50 Hz command transmission with frame dropping if tracking lags
  \item \textbf{UI thread:} 30 FPS rendering with overlays; handles user input (ROI selection, mode switching, graceful exit)
\end{enumerate}

Synchronization employs std::mutex for queue access, std::condition\_variable for blocking waits, and std::atomic for flag variables (stop signal, tracker ready flag).

\section{Detection Pipeline}

\subsection{Car Detection}

Motion-based detection leverages MOG2 background subtraction, more robust to color variation than HSV-based detection. The pipeline follows:

\begin{enumerate}
  \item Convert frame to grayscale
  \item Apply MOG2 with history=500 frames, variance threshold=16
  \item Threshold motion mask at 80 (aggressive cutoff to isolate motion)
  \item Morphological operations: open (3×3 kernel), close (5×5 kernel), dilate (3×3 kernel)
  \item Extract contours; filter by area (20--30000 px), bounding box aspect ratio (0.35--2.85), solidity (>0.45)
  \item Score candidates: $\text{score} = \text{distance\_from\_last} - 0.0008 \times \text{area}$
  \item Return centroid and bounding box of highest-scoring candidate
\end{enumerate}

Color-based fallback applies six HSV red ranges to capture hue variations, combines masks, and applies identical filtering logic. Priority ensures motion detection runs first; only on detection failure does color detection activate.

\subsection{Boundary and Centerline Detection}

Boundary detection builds track-aware guidance maps:

\begin{figure*}[t]
    \centering

    \caption{Track perception pipeline}
    \label{fig:pipeline}
\end{figure*}

\begin{enumerate}
  \item Convert frame to HSV
  \item Generate red mask: in Range(hsv, [0,70,50]--[12,255,255]) $\cup$ inRange(hsv, [168,70,50]--[179,255,255])
  \item Compute convex hull from red pixels; create ROI mask via fillConvexPoly
  \item Generate white mask: in Range(hsv, [0,0,180]--[179,70,255]) $\cap$ ROI
  \item Combine red + white as obstacles; apply morphological open/close/dilate
  \item Flood fill from image border to identify free space
  \item Extract largest connected component; fill small holes (<100 pixels)
  \item Apply distance transform on free space; extract skeleton via morphological thinning
  \item Smooth centerline via Gaussian kernel (σ=3)
\end{enumerate}

Centerline extraction runs every 30 frames to balance accuracy and computational cost. The resulting path provides steering targets for pure pursuit control.

\section{Control Strategy}

\subsection{Ray Casting Mode}

When centerline is unavailable or corrupted, ray casting provides fallback guidance. Given car position and heading (estimated from motion vector):

\begin{equation}
\text{heading} = \arctan2(\Delta y, \Delta x) \times 180 / \pi
\end{equation}

Three rays are cast at absolute angles: $\text{heading} -60\degree$, 
$\text{heading}$, $\text{heading} + 60\degree$. Each ray propagates from car center until hitting image boundary or black pixel (threshold $<50$). If closest ray distance $d_{\min} < 100$ pixels (evasive threshold), speed reduces to half default and steering commands counter the closest obstacle.

\subsection{Pure Pursuit Control}

When centerline is available, the car tracks the nearest point on the centerline:

\begin{equation}
\text{offset} = \text{distance\_to\_nearest\_centerline\_point}
\end{equation}

Proportional steering applies: $\text{steering} = k_p \times \text{offset}$, where $k_p=1$ (tuned via config.json). Left/right commands are then clamped to steering limit (default: 30).

Speed modulation reduces speed on sharp turns: $\text{speed} = \text{default\_speed} \times (1 - 0.3 \times |\text{steering\_strength}|)$.

\section{Implementation Details}

\subsection{Key Technologies}

\begin{itemize}
  \item \textbf{C++17:} Standard library, templates for type safety and zero-cost abstraction
  \item \textbf{OpenCV 4.x:} Image processing, tracking, and morphological operations
  \item \textbf{SimpleBLE:} Cross-platform BLE library with automatic device enumeration
  \item \textbf{nlohmann\_json:} Configuration parsing and validation
  \item \textbf{CMake:} Portable build system supporting macOS, Linux, Raspberry Pi
\end{itemize}

\subsection{Configuration System}

All parameters reside in \texttt{config/config.json} with reloadable settings (restart required). Key parameters:

\begin{lstlisting}

             "height": 576, "fps": 30},
  "ble": {"device_mac": "f9:af:3c:e2:d2:f5",
    "characteristic_uuid": 
    "6e400002-b5a3-f393-e0a9-e50e24dcca9e"},
  "boundary": {"black_threshold": 50,
    "ray_max_length": 200, "default_speed": 20,
    "steering_limit": 1},
  "ui": {"show_window": true, 
    "command_rate_hz": 50}
}
\end{lstlisting}

This approach decouples tuning from compilation, enabling rapid iteration on thresholds, speeds, and control gains without rebuilding.

\subsection{Performance Optimizations}

\begin{itemize}
  \item \textbf{Frame rate limiting:} Tracking thread enforces 15 FPS to balance responsiveness and CPU load. Idle threads sleep rather than spin.
  \item \textbf{Queue management:} Circular buffer limits memory footprint; BLE thread drops frames if tracking thread falls behind.
  \item \textbf{Lazy centerline update:} Centerline rebuilt every 30 frames (500 ms) unless track changes detected.
  \item \textbf{ROI-based processing:} Ray casting operates within convex hull of red pixels, avoiding unnecessary boundary checks.
\end{itemize}

\section{Testing and Validation}

\subsection{Functional Testing}

\textbf{Car Detection:} Validated on 50 video clips (lighting variation 200--5000 lux) with manual bounding box ground truth. Motion detection achieved 95\% recall; color detection fallback improved recovery on tracking loss from 40\% to 85\%.

\textbf{BLE Communication:} Continuous transmission over 60 minutes showed 99.8\% command delivery (1 drop per 500 commands typical). Reconnection on link loss occurs within 2 seconds.

\textbf{Path Following:} On a closed oval track (3 m perimeter, 0.5 m wide), the car completes 10 autonomous laps with average steering oscillation of ±8° around centerline.

\subsection{Performance Metrics}

On Raspberry Pi~4 (2.4 GHz quad-core ARM):
\begin{itemize}
  \item Camera-to-command latency: 85--120 ms (target: <150 ms)
  \item Tracking frame rate: 14.8--15.2 FPS (stable)
  \item BLE command rate: 50.1 Hz (±0.2 Hz, jitter <50 ms)
  \item Peak CPU utilization: 78\% (all threads)
  \item Memory footprint: 185 MB (excluding frame buffers)
\end{itemize}

\textbf{Field Test Results (45 second autonomous run):}
\begin{itemize}
  \item Total program execution time: 45 seconds
  \item Total BLE commands transmitted: 2250 (50 Hz rate maintained)
  \item Average BLE send latency: 0.85 ms per command
  \item Total accumulated BLE latency: 1912 ms
  \item Car completed: $\approx 3$ full laps on 540 cm track
  \item Average lap time: $\approx 15$ seconds at 40\% speed setting
\end{itemize}

\section{Challenges and Mitigations}

\textbf{Tracking Loss Under Blur:} Motion-based detection fails on rapid acceleration or track banking angles exceeding 15°. Mitigation: color-based fallback activates within 100 ms; manual ROI selection recovers in 2 seconds.

\textbf{Lighting Variation:} HSV thresholds require seasonal tuning (sunny vs. overcast). Mitigation: configuration allows per-session threshold updates without recompilation. Adaptive illumination correction (histogram equalization) reduces threshold sensitivity.

\textbf{BLE Interference:} Dense WiFi environments cause packet loss (0.2--2\%). Mitigation: command redundancy (repeated sends if previous sent >50 ms ago) and simple retry logic.

\textbf{Computational Budget:} Centerline extraction (distance transform) occupies 30--40\% CPU. Mitigation: reduce extraction frequency or downscale for embedded deployment.

\section{Conclusions}

This work demonstrates that production-grade autonomous control is achievable with minimal computational resources and a single camera. The system successfully navigates a 540~cm track for multiple laps using only motion-based detection and centerline-guided path following. Key design choices MOG2 background subtraction for robustness, HSV boundary detection for path guidance, and 50~Hz BLE communication, enable stable autonomous operation with 0.85~ms command latency.

The 45-second field test validated end-to-end functionality: 2250 BLE commands delivered without loss, stable 15~FPS tracking, and smooth steering control at 40\% speed (0.22~m/s). The modular C++17 architecture ports seamlessly across macOS, Linux, and Raspberry Pi, supporting both development and embedded deployment.

\section{Future Work}

Near-term improvements include: (1) dynamic speed modulation based on track curvature (currently fixed 40\%), (2) learning-based steering refinement via recorded trajectories, (3) sensor fusion with IMU for improved stability on track banking, and (4) multi-car coordination for formation control.

Long-term research directions involve: semantic segmentation for off-track obstacle detection, SLAM-based global localization for arbitrary environments, and hybrid control strategies combining vision-based and model-predictive approaches.
